# 2. 你们最适合的赛道
你们应该选：
# 赛道 A：Physical AI
更具体是：
> A.01 具身智能：机械臂、人形机器人、VLA、物理世界模型
原因很简单：你们有 G1 轮式双臂机器人、灵巧手、相机、打印机、补光灯、手机夹具、机械结构改装，这些都属于 Physical AI 的硬件实物演示。
不要选 AI Agent 赛道作为主赛道。
虽然你们有“摄影智能体调度器”，但你们的核心竞争力不是纯软件 Agent，而是：
> AI 走出屏幕，控制真实机器人完成拍照服务。
这句话正好贴合赛道 A 的主题：“让 AI 走出屏幕，触摸物理世界”。
你们可以在项目里说：
> 我们的上层调度采用 Agent 架构，但项目归属于 Physical AI，因为最终目标是让 Agent 调度真实机器人、相机、灵巧手、打印机完成物理服务闭环。
这样最稳。

# 3. 你们能冲刺的奖项
从规则看，评分大概是：
> 技术与产品 40% + 创业潜力 25% + 勤奋度 20% + 人气与传播 15%
所以你们不能只冲技术奖。你们可以同时冲这些：
## 第一目标：Physical AI 赛道奖
理由：
- 有真实 G1 机器人；
- 有机械臂/灵巧手；
- 有多模态感知；
- 有真实物理交互；
- 有完整场景闭环；
- 比单纯 App/Agent 更容易在现场吸引注意。
但前提是：现场必须真跑。
评分细则里“现场 Demo 完成度 12 分”很重要，评委会看你是真跑还是人工视频。
## 第二目标：最佳产品设计奖
这个你们很有希望。
你们不是做一个抽象论文项目，而是一个很明确的产品：
> 面向毕业典礼、会展、景区、商场快闪的具身摄影服务机器人。
只要你们把服务流程、用户体验、商业模式、照片交付、隐私策略讲完整，就很适合“最佳产品设计”。
## 第三目标：最佳人气奖
这也可以冲。
拍照机器人天然适合拉人互动。你们现场可以让观众：
- 让机器人帮忙拍照；
- 扫码取图；
- 拿一张打印照片；
- 投票；
- 小红书传播。
如果你们做得比“技术项目”更好玩，能拿人气分。
## 第四目标：最佳技术创新奖
这个能冲，但别把希望完全压在这里。
因为你们的底层算法不一定是原创研究，更多是系统集成和具身服务闭环。
如果要冲技术创新，必须强调：
> 机器人不是固定拍照亭，而是通过视觉构图、机械臂调整、灵巧手触屏、任务调度 Agent，实现“摄影行为的具身化”。

## 7.8：拿到机器人当天
这一天非常关键。你们要做的不是“开始幻想功能”，而是立刻验证硬件边界。
必须测试：
1. G1 SDK 能不能控制底盘；
2. G1 SDK 能不能控制左臂到指定姿态；
3. G1 SDK 能不能控制右臂/灵巧手；
4. 头部摄像头能不能读；
5. Insta360 是否能被 Jetson/电脑读取；
6. 机器人上是否有可用 USB/电源；
7. 手机夹具能不能装上法兰；
8. 右手点击手机屏幕是否有电容触控效果；
9. 打印机能否接收图片打印；
10. 补光灯能否固定和供电；
11. 机器人运行时线缆会不会缠绕；
12. 有无急停、安全限制、速度限制。
7.8 晚上必须得出一个结论：
> 哪些是真的能做，哪些只能放 PPT，哪些必须砍掉。

# 6. 另一个拍照机器人是什么定位
“IP有界、快乐无界”这个队伍，按他们那句话：
> “我们想用影石相机和我们的熊猫机器人一起做一个拍照机器人，可以给大家拍照留念，拍照立等可取。”
他们大概率是：
# 熊猫 IP 互动拍照亭 / 萌宠打卡机器人
他们的强项不是技术闭环，而是：
- 熊猫形象；
- 可爱；
- 活动打卡；
- 拍照留念；
- 立等可取；
- 观众传播；
- 人气投票。
他们的项目可能很适合拿：
- 最佳人气奖；
- 最佳产品设计奖；
- 商业创意奖；
- 传播分。
但他们未必适合拿技术创新奖，除非他们也做了很强的自动构图、交互、导航或机器狗控制。
你对“熊猫机器人”的猜测可能是对的：
它可能是机器狗或轮式底盘加熊猫外壳，也可能只是一个熊猫造型的移动装置。核心是 IP 化，而不是摄影能力。

# 7. 你们怎么规避和他们撞题
你们一定要避免以下表述：
> “我们也是做拍照机器人，给大家拍照留念，立等可取。”
这句话一说，你们就和他们撞了，而且他们有熊猫 IP，可能现场更讨喜。
你们应该改成：
> 我们不是 IP 拍照装置，而是一个能执行摄影服务流程的移动双臂具身摄影师。
你们的差异化要打四点。
## 差异 1：他们是 IP 合影，你们是具身执行
他们可能是“熊猫和你合影”。
你们是“机器人替你完成摄影师动作”。
你们要展示：
- 机械臂调整相机角度；
- 灵巧手点击手机；
- 机器人语音引导；
- 构图分析；
- 照片交付；
- 多设备调度。
## 差异 2：他们是固定拍照，你们是摄影流程
他们可能是：
```plaintext
站好 → 拍照 → 打印

```
你们要做：
```plaintext
接待 → 选择模式 → 找角度 → 构图判断 → 指导站位 → 拍照 → 选片 → 打印/下载

```
你们卖的是“流程闭环”。
## 差异 3：他们是情绪价值，你们是服务能力
他们的关键词是：
> 熊猫、快乐、IP、打卡。
你们的关键词是：
> Physical AI、具身智能、双臂操作、多模态感知、Agent 调度、服务机器人半解决方案。
## 差异 4：他们是“被拍对象”，你们是“摄影师”
熊猫机器人很可能是大家想跟它合影。
你们的 G1 应该是：
> 它不只是被拍，而是主动帮人拍。
这句话很关键。

# 9. 是否可以举报他们“非本赛期工作内容 / 预制菜”
我的建议：不要轻易举报，除非你有明确证据。
原因有三个。
第一，黑客松通常允许使用赛前已有硬件、已有开源项目、已有机器人平台。
你们自己也不是从零造 G1，也不是从零造灵巧手，也会提前 7.6 筹备、7.8 拿机器人。你如果举报他们“熊猫机器人不是赛期做的”，对方也可以反问你们：“G1、夹具设计、方案是不是也赛前准备了？”
第二，规则看起来重点是：
> 赛期内的项目开发、系统集成、代码提交、Demo 完成度。
不是说硬件必须 48 小时内从零制造。Physical AI 赛道本来就会用现成机器人、机械臂、开发板、传感器。
第三，举报容易伤害你们队伍形象。
评委和主办方更喜欢正向竞争，不喜欢参赛队互相攻击。除非对方直接拿成熟商业产品冒充 48 小时开发成果，否则不建议上来就举报。
更好的方式是：
> 向主办方中立询问规则，而不是点名举报。
你可以这样问：
> “请问本次 Physical AI 赛道是否允许使用赛前已有机器人本体、外观 IP 或已有硬件系统？评审时是否主要看 48 小时内新增的软件、集成、交互和 Demo 完成度？我们想确认边界，避免不必要的误解。”
如果主办方说可以用既有硬件，那就别纠结。
如果主办方明确说“不能用赛前完成的完整项目”，你再看他们是否明显违规。
可以举报的情况只有这些：
- 他们提交的 GitHub 基本没有赛期新增内容；
- Demo 是赛前完整产品，赛期只是拿来展示；
- 对外宣称 48 小时开发，但实际核心系统早已完成；
- 使用不允许的商业闭源成品并伪装成自研；
- 违反比赛安全/硬件使用规则；
- 侵犯 IP 或肖像授权。
但“熊猫外壳/机器人本体赛前已有”本身不一定违规。

# 10. 你们之前讨论里遗漏的东西
你们之前想法很丰富，但有几个关键遗漏。
## 遗漏 1：安全策略
你们涉及：
- 机器人靠近人；
- 机械臂伸出；
- 灵巧手点击手机；
- 用户把手机交给机器人；
- 打印机发热；
- 补光灯；
- 现场人群围观。
PPT 和 Demo 里必须讲安全：
- 机械臂低速模式；
- 固定安全距离；
- 人手交接区域；
- 手机夹具防跌落；
- 急停；
- 不做高速动作；
- 不在密集人群中自由巡航；
- 拍照时机器人静止。
这能让评委放心。
## 遗漏 2：隐私与授权
拍照项目一定要讲隐私。
建议写：
> 默认本地处理，不上传云端；用户扫码下载后可删除；未经授权不用于训练；用户手机模式下照片只保存在用户手机内。
现场最好贴一张小牌：
> 体验即表示同意拍摄用于本次 Demo；如需删除请联系工作人员。
## 遗漏 3：失败兜底
机器人一定会有失败风险。你们要设计兜底。
比如：
- 机械臂失败 → 使用固定机位相机拍；
- 打印失败 → 二维码下载；
- 二维码失败 → 本地文件夹展示；
- 导航失败 → 人工移动/固定点位；
- 灵巧手点击失败 → 蓝牙快门/键盘快捷键；
- 人脸检测失败 → 直接拍照。
黑客松里兜底不是丢人，兜底是工程能力。
## 遗漏 4：拍摄质量标准
你们不能只说“拍照”。
要定义什么叫“拍得好”：
- 人是否完整入镜；
- 人脸是否清晰；
- 是否闭眼；
- 是否过曝/欠曝；
- 人物是否居中；
- 背景是否干净；
- 是否有阴阳脸；
- 是否模糊；
- 是否符合活动纪念照风格。
这样才有 AI 味。
## 遗漏 5：电子交付方案
你们已经意识到缺屏幕，这个必须尽快解决。
最现实方案：
- 外接小屏幕显示二维码；
- 或笔记本屏幕显示二维码；
- 或机器人旁边放一个平板；
- 或打印照片 + 小票二维码；
- 或局域网网页让用户扫码看照片。
不要依赖 G1 胸前屏幕接口。拿不到就立刻外接。
## 遗漏 6：现场排队与服务时长
一次服务必须控制在 30–60 秒。
如果你们流程太长，观众和评委都等不了。
建议：
- Demo 模式：30 秒；
- 完整模式：60–90 秒；
- 路演展示：只展示最关键 1 次动作。
## 遗漏 7：路演叙事
你们不能把所有硬件堆给评委。
6 分钟路演建议结构：
1. 30 秒：痛点，活动现场没人帮拍/拍不好/取图麻烦；
2. 60 秒：解决方案，G1 具身摄影服务机器人；
3. 90 秒：现场 Demo 或视频，接待、构图、拍照、交付；
4. 60 秒：技术架构，视觉 + Agent + 机械臂 + 相机 + 打印；
5. 60 秒：商业模式，ToB 活动租赁 + ToC 打印下载；
6. 30 秒：和普通拍照亭/熊猫 IP 的区别；
7. 30 秒：未来计划。
## 遗漏 8：团队分工
你们需要立刻按角色分工，不要大家都在讨论。
推荐：
- Hardware Lead：夹具、相机、打印机、补光灯、线缆、安全；
- Robot Control：G1 SDK、机械臂姿态、灵巧手动作；
- Vision/AI：相机读取、人脸检测、构图评分、选片；
- Agent/App：流程状态机、网页、二维码、语音；
- Product/Pitch：PPT、README、小红书、Demo 视频、商业模式。
如果只有 3–4 人，就合并角色，但必须有人专门负责材料。
## 遗漏 9：GitHub 和 README
GitHub 是必交项，而且影响代码质量分。
README 至少要有：
- 项目简介；
- 系统架构图；
- 硬件清单；
- 软件模块；
- 安装运行方式；
- Demo 流程；
- 隐私说明；
- 商业化说明；
- 团队成员分工；
- 现场照片/视频链接。
不要等最后写。
